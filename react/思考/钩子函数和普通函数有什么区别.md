## React 钩子函数 vs 普通函数

### 最核心的区别

钩子函数能**接入 React 的内部系统**（状态、生命周期、上下文），普通函数不能。

```jsx
// 普通函数 —— 只是计算，和 React 没有任何关系
function add(a, b) {
  return a + b;
}

// 钩子函数 —— 接入了 React 内部的状态系统
function useCount() {
  const [count, setCount] = useState(0); // 这行普通函数里写了会报错
  return [count, setCount];
}
```

---

### 本质差异对比

| | 普通函数 | 钩子函数 |
|---|---|---|
| 命名 | 随意 | 必须 `use` 开头 |
| 调用位置 | 任何地方 | 只能在组件/钩子顶层 |
| 能用其他Hook | ❌ | ✅ |
| 有副作用能力 | ❌ | ✅ |
| 和渲染绑定 | ❌ | ✅ 每次渲染都会执行 |
| React 追踪 | ❌ | ✅ |

---

### 为什么钩子有这些限制？

React 内部用一个**链表**按顺序记录所有 Hook 的状态：

```
渲染时 Hook 调用顺序：
useState   → 链表节点1 { value: 0 }
useEffect  → 链表节点2 { deps: [] }
useRef     → 链表节点3 { current: null }
```

React 靠的是**调用顺序**来对应每个 Hook 的状态，所以：

```jsx
// ❌ 不能在条件里调用——顺序会乱掉
if (someCondition) {
  useState(0); // 某次渲染跳过了，链表对不上，崩溃
}

// ❌ 不能在普通函数里调用——React 不知道是哪个组件的
function helper() {
  useState(0); // React 找不到对应的组件上下文
}
```

---

### 直观例子说明差异

```jsx
// ❌ 普通函数版本 —— 点击没有效果，组件不会重渲染
function getCount() {
  let count = 0; // 每次调用都重置，React 完全不知道这个变量
  const add = () => count++;
  return { count, add };
}

// ✅ 钩子版本 —— React 托管状态，变化驱动重渲染
function useCount() {
  const [count, setCount] = useState(0); // React 记住了这个值
  const add = () => setCount(c => c + 1); // 触发重渲染
  return { count, add };
}
```

---

### use 开头不是规范，是必须

`use` 前缀不只是命名约定，而是 React 和 lint 工具识别钩子的唯一方式：

```jsx
// React 看到 use 开头 → 按钩子规则检查和处理
// React 看不到 use 开头 → 当普通函数，不做任何 Hook 相关处理

function getData() {       // 普通函数，里面写 useState 会出问题
  useState(0);             // ⚠️ eslint 报错，运行时也可能出错
}

function useGetData() {    // 钩子，可以安全使用其他 Hook
  useState(0);             // ✅
}
```

---

### 一句话总结

> 普通函数是**你的代码**，钩子函数是**你的代码 + React 系统的入口**。  
> `use` 开头告诉 React："请帮我管理这里面的状态和副作用"。
